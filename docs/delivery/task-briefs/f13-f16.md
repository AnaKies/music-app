## F13. Case Edit And Reset

### Frontend-24

Objective:
Add the edit-case action UI.

Context:
Users must be able to adjust case constraints without abandoning the current architecture path.

Deliverable:
An edit action entry point from relevant case-aware screens.

Dependencies:
Case UI baseline and case read endpoint.

Acceptance criteria:

- edit action is visible in the expected contexts
- action routes back into the case/interview editing path
- current case context remains clear

Out of scope:
Actual persistence of edits.

Suggested verification:
Navigation and UI tests.

### Frontend-25

Objective:
Add the reset-case action UI.

Context:
Reset is explicit user intent and should not be hidden or accidental.

Deliverable:
A reset action entry point with appropriate user-facing framing.

Dependencies:
Case UI baseline.

Acceptance criteria:

- reset action is reachable
- action is clearly distinct from edit
- accidental triggering is reduced through normal UI safeguards

Out of scope:
Backend reset semantics.

Suggested verification:
UI interaction and confirmation-flow tests.

### Frontend-26

Objective:
Implement the return path from recommendation back to interview.

Context:
Case edits may invalidate recommendations and require renewed interview work.

Deliverable:
A clear return flow from recommendation review back into interview or case editing.

Dependencies:
Backend case-edit/reset behavior and recommendation stale-state handling.

Acceptance criteria:

- user can move back intentionally
- transition preserves case identity
- stale or outdated recommendation state is not silently reused

Out of scope:
Recommendation regeneration logic itself.

Suggested verification:
Flow navigation test.

### Backend-42

Objective:
Implement reset semantics.

Context:
Reset should clear or restart case-specific state without creating ambiguous lifecycle behavior.

Deliverable:
Backend reset behavior aligned with the case contract.

Dependencies:
Case persistence model.

Acceptance criteria:

- reset affects the intended case only
- resulting case status is explicit
- downstream recommendation state reacts correctly

Out of scope:
Archive behavior.

Suggested verification:
Case lifecycle integration tests.

### Backend-43

Objective:
Implement case-edit persistence.

Context:
Edits must update stored constraints in a controlled way.

Deliverable:
Persistent update behavior for case constraints.

Dependencies:
Case persistence model and confirmed/inferred separation rules.

Acceptance criteria:

- edited confirmed fields persist correctly
- affected downstream states are marked as needing refresh where required
- unrelated cases remain untouched

Out of scope:
Interview question generation.

Suggested verification:
Persistence and state-transition tests.

### AI-12

## F14. Stale Recommendation Handling

### Backend-44

Objective:
Mark recommendation snapshots as stale after relevant case edits.

Context:
Recommendations must remain tied to the case state they were generated from.

Deliverable:
A backend stale-flag update path for affected recommendation snapshots.

Dependencies:
Case edit persistence and recommendation persistence.

Acceptance criteria:

- stale flag is set only when relevant case changes occur
- stale information persists durably
- unrelated recommendations are not marked stale accidentally

Out of scope:
Frontend warning rendering.

Suggested verification:
State-transition tests.

### Backend-45

Objective:
Block stale recommendation reuse.

Context:
Stale recommendations must not drive deterministic transformation.

Deliverable:
Backend enforcement preventing stale recommendation execution.

Dependencies:
Backend-44 and transformation validation.

Acceptance criteria:

- stale recommendations are rejected for transformation
- failure or follow-up-needed behavior is typed
- fresh recommendations remain usable

Out of scope:
Recommendation regeneration itself.

Suggested verification:
Validation regression tests.

### Frontend-27

Objective:
Render stale recommendation state.

Context:
Users must see when existing recommendations are no longer safe to use.

Deliverable:
UI rendering for stale recommendation state.

Dependencies:
Stale flag in the score or recommendation read model.

Acceptance criteria:

- stale status is visible
- stale state is not confused with normal readiness
- user understands that selection cannot continue normally

Out of scope:
Backend stale logic.

Suggested verification:
UI state rendering tests.

### Frontend-28

Objective:
Render regeneration-required guidance.

Context:
Stale state should point the user toward the next safe action.

Deliverable:
A visible guidance or action area explaining that recommendations must be regenerated.

Dependencies:
Frontend-27 and stale-state contract.

Acceptance criteria:

- user sees what must happen next
- stale state does not present a misleading continue action
- guidance is concise and non-technical

Out of scope:
Actual regeneration route implementation if handled elsewhere.

Suggested verification:
UI behavior tests.

### Test-44

Objective:
Add a stale-flag regression test.

Context:
Relevant case edits must always invalidate old recommendation snapshots.

Deliverable:
A regression test around stale-flag updates.

Dependencies:
Backend-44.

Acceptance criteria:

- relevant changes mark stale correctly
- irrelevant changes do not over-trigger staleness
- stale flag remains visible in read models

Out of scope:
Frontend rendering.

Suggested verification:
Persistence and read-model regression test.

### Test-45

Objective:
Add a test that stale recommendations cannot be transformed.

Context:
This is a critical architecture and safety boundary.

Deliverable:
A regression test blocking stale recommendation execution.

Dependencies:
Backend-45.

Acceptance criteria:

- stale recommendation transformation is rejected
- failure behavior is typed
- fresh recommendations still pass

Out of scope:
UI stale messaging.

Suggested verification:
Backend validation test.

### Test-46

Objective:
Add a UI test for stale-state communication.

Context:
The stale state must be visible and understandable in the interface.

Deliverable:
A UI test for stale-state rendering and guidance.

Dependencies:
Frontend-27 and Frontend-28.

Acceptance criteria:

- stale markers render
- regeneration guidance appears
- normal selection flow is unavailable in stale state

Out of scope:
Regeneration backend behavior.

Suggested verification:
Screen-level frontend test.

### Safety-7

Objective:
Review that stale recommendations are never silently reused.

Context:
Stale reuse would violate both trust and deterministic execution boundaries.

Deliverable:
A safety review confirming stale reuse is blocked.

Dependencies:
Backend-45 and stale read-model visibility.

Acceptance criteria:

- stale recommendations cannot be consumed silently
- UI does not mask stale state as normal readiness
- review findings are documented if any unsafe path remains

Out of scope:
General recommendation quality review.

Suggested verification:
Safety checklist and payload review.

## F15. Retryable Failure Recovery

### Backend-46

Objective:
Finalize typed failure classes.

Context:
Retry behavior and user-facing failure state depend on typed failures rather than generic exceptions.

Deliverable:
A finalized failure-class model for parsing, recommendation, transformation, and export failures.

Dependencies:
Observability failure model.

Acceptance criteria:

- failure classes are explicit
- each class maps to a known workflow step
- generic unknown failures are minimized

Out of scope:
Frontend retry rendering.

Suggested verification:
Failure-model tests.

### Backend-47

Objective:
Expose retryable failure flags.

Context:
The UI must only offer retry when recovery is actually safe.

Deliverable:
Retryability metadata attached to failure states.

Dependencies:
Backend-46 and read-model normalization.

Acceptance criteria:

- retryable is explicit
- unsafe failures are not marked retryable
- flags are stable across endpoints

Out of scope:
Retry button behavior.

Suggested verification:
Failure-read-model tests.

### Backend-48

Objective:
Generate `safeSummary` values for failures.

Context:
Failure communication should remain concise and presentation-safe.

Deliverable:
User-facing safe summaries for failure states.

Dependencies:
Backend-46 and read-model normalization.

Acceptance criteria:

- summaries are short and understandable
- raw diagnostics are not leaked
- summaries exist for major failure classes

Out of scope:
Detailed operator diagnostics.

Suggested verification:
Read-model safety tests.

### Frontend-29

Objective:
Render retry controls only for retryable states.

Context:
Retry is a safety-sensitive action and must not appear universally.

Deliverable:
UI behavior that shows retry only when the backend marks the path as retryable.

Dependencies:
Backend-47 and status read models.

Acceptance criteria:

- retry is visible only on retryable states
- non-retryable failures do not show the same action
- retry placement is clear and consistent

Out of scope:
Backend retry semantics.

Suggested verification:
UI state tests.

### Frontend-30

Objective:
Distinguish retryable and non-retryable failure UI.

Context:
Users must understand whether they can safely act or need another path.

Deliverable:
Distinct UI treatment for retryable versus non-retryable failures.

Dependencies:
Backend-47, Backend-48, and status rendering baseline.

Acceptance criteria:

- the two failure modes look behaviorally different
- retry guidance is concise
- failure states do not look like normal completion

Out of scope:
Backend failure-class logic.

Suggested verification:
Frontend rendering tests.

### Test-47

Objective:
Add a retryable-failure test.

Context:
Retryable failure behavior must stay stable across refactoring.

Deliverable:
An automated test covering one or more retryable failure cases.

Dependencies:
Backend-47 and Frontend-29 where UI is involved.

Acceptance criteria:

- retryable flag is set correctly
- retry UI appears only in the expected case
- typed failure remains visible

Out of scope:
Non-retryable path.

Suggested verification:
Backend and UI integration test.

### Test-48

Objective:
Add a non-retryable-failure test.

Context:
Unsafe retries should not be offered.

Deliverable:
A test covering at least one non-retryable failure case.

Dependencies:
Backend-47 and Frontend-30 where UI is involved.

Acceptance criteria:

- retry is absent
- failure state is still clear
- no accidental recovery path is offered

Out of scope:
Retry execution implementation.

Suggested verification:
Backend and UI regression test.

### Test-49

Objective:
Add a UI test for retry visibility.

Context:
Retry visibility is one of the clearest recoverability signals in the product.

Deliverable:
A frontend test for retry visibility rules.

Dependencies:
Frontend-29 and Frontend-30.

Acceptance criteria:

- retry appears only when expected
- failure summaries remain readable
- completion and retryable failure states stay distinct

Out of scope:
Cloud restart semantics.

Suggested verification:
Screen-level UI test.

### Safety-8

Objective:
Review that retry is not offered on unsafe recovery paths.

Context:
This is a safety boundary, not only a convenience detail.

Deliverable:
A safety review of retry visibility and failure metadata.

Dependencies:
Backend-47, Backend-48, and frontend failure rendering.

Acceptance criteria:

- unsafe retries are absent
- retryable semantics align with documented failure classes
- any unsafe ambiguity is documented

Out of scope:
Infrastructure retry configuration.

Suggested verification:
Safety checklist against failure payloads and UI states.
