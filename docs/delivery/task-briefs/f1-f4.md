## F1. Case Entry

### Frontend-1

Objective:
Create the initial case-entry screen scaffold.

Context:
This screen is the first user-facing entry into the MVP flow and must support both continuation and new-case creation later.

Deliverable:
A working screen shell with placeholder regions for default case, case list, and new-case action.

Likely implementation area:
`src/app/` and `src/features/cases/`

Dependencies:
None inside the feature.

Acceptance criteria:

- the route or page exists
- the screen renders without backend data
- there are dedicated sections for suggested case, other cases, and new-case creation

Out of scope:
Real backend data wiring and case ranking logic.

Suggested verification:
Basic render test and manual navigation check.

### Frontend-2

Objective:
Render the list of existing cases from backend data.

Context:
The UI must show reusable cases without inventing case fields outside the approved contract.

Deliverable:
A rendered case list based on the case summary response.

Likely implementation area:
`src/features/cases/` and shared list/card components

Dependencies:
Case contract must be stable.

Acceptance criteria:

- cases render from fetched data
- empty list behavior is handled
- displayed fields match the approved case summary

Out of scope:
Default highlighting and case creation action.

Suggested verification:
UI test for populated and empty case lists.

### Frontend-3

Objective:
Highlight the most recently used active case as the default suggestion.

Context:
The architecture requires a recommended default case rather than a flat undifferentiated list.

Deliverable:
Visual emphasis for the recommended case and calmer presentation of the remaining cases.

Likely implementation area:
`src/features/cases/` and shared workspace/status primitives

Dependencies:
Frontend-2 and stable case ordering input.

Acceptance criteria:

- exactly one suggested case is highlighted when available
- archived or inactive cases are not treated as the default suggestion
- remaining cases stay selectable

Out of scope:
Changing backend sort order.

Suggested verification:
UI test for default-case emphasis with multiple cases.

### Frontend-4

Objective:
Wire the "create new case" action.

Context:
Users must be able to start a new flow when no current case should be reused.

Deliverable:
A working action that starts the new-case path or case-creation flow.

Likely implementation area:
`src/features/cases/` and app routing

Dependencies:
Case-entry screen scaffold exists.

Acceptance criteria:

- the action is visible and clickable
- it routes or transitions into the new-case flow
- it does not break the existing case list state

Out of scope:
Interview implementation.

Suggested verification:
Navigation test and manual flow check.

### Backend-1

Objective:
Implement the basic `POST /cases` route.

Context:
This endpoint starts a new transposition case and must follow the approved case contract.

Deliverable:
A route that accepts valid case creation input and returns the expected summary response.

Likely implementation area:
`src/backend/api/`, `src/backend/schemas/`, `src/backend/services/cases/`

Dependencies:
None beyond the approved interface contract.

Acceptance criteria:

- valid requests create a case
- response shape matches the approved contract
- invalid input is rejected with typed validation behavior

Out of scope:
Reset semantics and archive behavior.

Suggested verification:
API contract test and service-level creation test.

### Backend-2

Objective:
Implement the basic `GET /cases/{id}` route.

Context:
Frontend state and later upload gating depend on a durable case read endpoint.

Deliverable:
A route returning case summary, active constraints, linked score count, and status.

Likely implementation area:
`src/backend/api/`, `src/backend/readmodels/`, `src/backend/services/cases/`

Dependencies:
Case persistence model exists.

Acceptance criteria:

- valid case IDs return the documented payload
- unknown case IDs return not found
- response fields remain additive and typed

Out of scope:
Derived recommendation state.

Suggested verification:
API read-contract test and not-found test.

### Backend-3

Objective:
Define the case-summary response schema.

Context:
Frontend and tests need one stable schema before list and detail rendering fan out.

Deliverable:
A typed schema for case summary payloads.

Likely implementation area:
`src/backend/schemas/`

Dependencies:
Approved interface contract only.

Acceptance criteria:

- schema contains only approved fields
- schema can be reused by both create and read endpoints
- optional and required fields are explicit

Out of scope:
Persistence implementation.

Suggested verification:
Schema serialization test.

### Test-1

### Test-2

Objective:

Context:
This test protects frontend and later flow logic from read-model drift.

Deliverable:

Dependencies:

### Test-3

Objective:

Dependencies:

## F2. Structured Interview Session

### AI-1

Objective:
Define the question object schema.

Context:
The interview flow must remain structured and renderable without relying on untyped free-form AI responses.

Deliverable:
A typed question schema that covers approved question types and optional help text.

Likely implementation area:
`src/backend/ai/` or shared AI schemas

Dependencies:
Approved interview contract.

Acceptance criteria:

- approved question types are represented
- required and optional fields are explicit
- schema is suitable for frontend rendering

Out of scope:
Follow-up decision logic.

Suggested verification:
Schema fixture tests.

### AI-2

Objective:
Define the answer payload schema.

Context:
Backend and frontend need one stable answer shape before question rendering and submission logic proceed.

Deliverable:
A typed answer schema aligned with the approved contract.

Dependencies:
Approved interview contract.

Acceptance criteria:

- required answer fields are explicit
- optional local validation state is supported without becoming mandatory
- schema rejects structurally invalid answers

Out of scope:
Persistence and routing.

Suggested verification:
Schema validation tests.

### AI-3

Objective:
Define follow-up rules for incomplete or ambiguous answers.

Context:
Low-quality interview interpretation must lead to clarification rather than silent guessing.

Deliverable:
Clear rules that describe when another question is required.

Dependencies:
AI-1 and AI-2.

Acceptance criteria:

- ambiguous answer conditions are listed
- follow-up output remains structured
- rules do not silently confirm missing information

Out of scope:
Frontend wording polish.

Suggested verification:
Fixture tests with incomplete answers.

### AI-4

Objective:
Define low-confidence interview behavior.

Context:
The architecture requires explicit low-confidence handling before confirmed constraints are updated.

Deliverable:
A documented and implementable low-confidence branch for interview classification.

Dependencies:
AI-1 and AI-2.

Acceptance criteria:

- low-confidence does not update confirmed constraints directly
- follow-up or blocked behavior is explicit
- output remains schema-constrained

Out of scope:
Recommendation confidence logic.

Suggested verification:
Evaluation fixtures for low-confidence input.

### Backend-4

Deliverable:

Dependencies:

Acceptance criteria:

- response aligns with the interview contract

Case readiness persistence.

Suggested verification:

Out of scope:
Question generation logic.

Suggested verification:
Read-route contract test.

### Backend-6

Objective:
Add interview-session persistence.

Context:
Interview flows span multiple answers and may need recovery or continuation.

Deliverable:
Persistent interview session storage tied to case progression.

Dependencies:
Interview schemas and route contract.

Acceptance criteria:

- answers persist across requests
- session completion state is stored
- session can be resumed safely

Out of scope:
Final case readiness logic.

Suggested verification:
Persistence and continuation tests.

### Backend-6a

Objective:
Add a provisional case-delete endpoint for local MVP cleanup.

Context:
Manual and automated MVP testing become noisy when stale non-suggested cases accumulate without any removal path.

Deliverable:
A temporary delete route that can remove a case from the local MVP flow.

Dependencies:
Stable case read and list behavior.

Acceptance criteria:

- an existing case can be deleted through a typed backend route
- unknown case IDs return not found
- the route is documented and clearly marked as provisional

Out of scope:
Final archive/reset lifecycle design from later case-management features.

Suggested verification:
Backend contract test for delete success and not-found behavior.
