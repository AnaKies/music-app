## F5. Score Parsing

### Backend-15

Objective:
Add a parser wrapper around the chosen MusicXML parsing approach.

Context:
The parser wrapper should isolate parser-specific details from the rest of the backend.

Deliverable:
A backend component that accepts a valid upload handoff and returns either canonical parse input or a typed parse failure.

Likely implementation area:
`src/backend/services/scores/` or parsing domain module

Dependencies:
Upload handoff contract from F4.

Acceptance criteria:

- parser wrapper accepts the expected input type
- parser-specific exceptions are not leaked directly upward
- failure mapping is possible from one place

Out of scope:
Canonical-score persistence.

Suggested verification:
Backend unit tests with parser fixtures.

### Backend-16

Objective:
Implement canonical-score mapping.

Context:
Downstream recommendation and transformation logic must not depend on raw upload format details.

Deliverable:
A mapping from parsed MusicXML structures into the canonical score model.

Dependencies:
Backend-15.

Acceptance criteria:

- canonical structure is produced for valid input
- required score elements are normalized consistently
- output is suitable for later context assembly

Out of scope:
Recommendation logic.

Suggested verification:
Parser-to-canonical mapping tests.

### Backend-17

Objective:
Persist parse-success state.

Context:
The system must expose score-processing progress through durable metadata.

Deliverable:
Persistent metadata showing parse success and score readiness for later steps.

Dependencies:
Backend-16.

Acceptance criteria:

- parse-success state is stored durably
- the correct score document is linked
- later read-model assembly can see the new state

Out of scope:
Recommendation generation.

Suggested verification:
Persistence integration tests.

### Backend-18

Context:

Dependencies:
