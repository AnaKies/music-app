## F9. Deterministic Transformation

### Backend-25

Objective:
Implement the `POST /transformations` route.

Context:
This route is the public boundary for starting deterministic score transformation after user selection.

Deliverable:
A route that validates input and starts the transformation flow using the approved contract.

Dependencies:
Recommendation selection contract and transformation request schema.

Acceptance criteria:

- valid requests are accepted
- invalid or incomplete input is rejected
- route stays separate from raw AI output handling

Out of scope:
Transformation engine internals.

Suggested verification:
Contract and validation tests.

### Backend-26

Objective:
Validate selected recommendation input.

Context:
Transformation should only run from a valid selected recommendation or equivalent validated target-range input.

Deliverable:
Validation logic for transformation requests.

Dependencies:
Recommendation persistence and transformation contract.

Acceptance criteria:

- unknown or stale recommendation input is rejected
- required fields are validated
- validation errors are typed

Out of scope:
Warning generation.

Suggested verification:
Service validation tests.

### Backend-27

Objective:
Create the transformation-engine scaffold.

Context:
The deterministic transformation engine needs a dedicated internal boundary before detailed logic expands.

Deliverable:
A transformation module or service scaffold with clear inputs and outputs.

Dependencies:
Transformation contract only.

Acceptance criteria:

- engine entry point exists
- validated input shape is explicit
- output shape supports warnings and result handoff

Out of scope:
Full transposition logic.

Suggested verification:
Unit test of engine entry boundary.

### Backend-28

Objective:
Implement range adaptation logic.

Context:
This is the main deterministic transformation behavior that applies the selected target range.

Deliverable:
Working deterministic adaptation logic against the canonical score.

Dependencies:
Backend-27 and canonical score availability.

Acceptance criteria:

- notes are adapted according to the selected range
- logic remains deterministic
- output stays compatible with export

Out of scope:
Export format generation.

Suggested verification:
Domain tests with canonical score fixtures.

### Backend-29

Objective:
Emit warnings when adaptation is not clean.

Context:
The transformation path must surface limitations rather than silently masking them.

Deliverable:
Structured warnings emitted by transformation when exact adaptation is problematic.

Dependencies:

Acceptance criteria:

- warning cases are identifiable
- warnings are structured enough for read models
- normal successful paths do not emit false warnings

Out of scope:

Suggested verification:
